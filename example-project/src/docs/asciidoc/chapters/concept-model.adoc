=== Concept Design

The `Concept` record embodies key principles from SNOMED CT:

==== Immutability

Records provide immutable data structures, ensuring Koncepts cannot be modified after creation:

[source,java]
----
// Once created, this Koncept cannot be changed
Concept diabetes = new Concept(
    73211009L,
    "Diabetes mellitus (disorder)",
    "Diabetes mellitus",
    "A metabolic disorder..."
);
----

==== Validation

Compact constructors enforce business rules:

[source,java]
----
public Concept {
    if (conceptId <= 0) {
        throw new IllegalArgumentException("Concept ID must be positive");
    }
    // ... additional validation
}
----

==== Factory Methods

Specialized factory methods ensure correct semantic tag application:

[source,java]
----
// Automatically adds (finding) semantic tag
Concept finding = Concept.clinicalFinding(
    38341003L,
    "Hypertension",
    "High blood pressure",
    "Elevated arterial blood pressure"
);
----

=== Koncept Relationships

In a full implementation, Koncepts are connected through formal relationships.
For example, koncept:Diabetes mellitus[] is related to both
koncept:Endocrine disorder[] and koncept:Metabolic disorder[] through
subsumption, while its definition includes existential restrictions
to koncept:Pancreatic structure[] and koncept:Hyperglycemia[].

=== Description Logic Representation

SNOMED CT Koncepts can be expressed in description logic notation.
The definition of koncept:Diabetes mellitus[uuid=73211009] is:

[source]
----
Diabetes mellitus ⊑ Endocrine disorder ⊓ Metabolic disorder ⊓
    ∃ finding-site.Pancreatic structure ⊓
    ∃ has-finding.Hyperglycemia
----

Where:
* `⊑` represents subsumption (is-a)
* `⊓` represents intersection (and)
* `∃` represents existential restriction (some)

=== Primitive vs. Fully Defined

[cols="1,3,3"]
|===
|Type |Definition |Example

|Primitive
|Necessary conditions only
|"A disease affecting the pancreas" (not sufficient — other diseases also affect pancreas)

|Fully Defined
|Necessary and sufficient conditions
|"A metabolic disorder with hyperglycemia, caused by insulin deficiency or resistance"
|===

The distinction is critical for automated classification.
A fully defined Koncept like koncept:Diabetes mellitus[uuid=73211009]
provides necessary _and_ sufficient conditions, enabling a DL reasoner
to infer its position in the hierarchy.  A primitive Koncept like
koncept:Disease[] provides only necessary conditions — subsumption
relationships must be asserted by human modelers.

The `isPrimitive()` method provides a simplified check:

[source,java]
----
if (concept.isPrimitive()) {
    // Handle incomplete definition
    // May need human review for classification
} else {
    // Fully defined — can be automatically classified
    // Description logic reasoner can infer relationships
}
----

=== Future Enhancements

The Koncept model could be extended with:

* Relationship representation
* Role groups for complex definitions
* Effective time and versioning
* Multiple language support
* Reference set membership
* Concrete domains (numeric values, strings)
