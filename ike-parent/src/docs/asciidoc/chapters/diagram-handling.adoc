== Diagram Handling

The pipeline supports PlantUML, GraphViz, and Mermaid diagrams. All diagram types are rendered server-side by Kroki -- no local CLI tools (mmdc, plantuml.jar, dot) are required.

IMPORTANT: **Prefer PlantUML and GraphViz over Mermaid.** PlantUML and GraphViz emit clean SVG that renders correctly across all backends without workarounds. Mermaid's SVG output relies on `<foreignObject>` elements for label rendering, which causes failures in Prawn and WeasyPrint and requires svgo post-processing for Prince, AH, and XEP. Use Mermaid only for HTML-only output or diagram types without PlantUML equivalents (Gantt, pie). See the `IKE-DIAGRAMS` standard for full editorial guidance.

=== Kroki Server-Side Rendering

Diagrams are delegated to a Kroki server configured in the parent POM:

[source,xml]
----
<diagram-server-url>${kroki.server.url}</diagram-server-url>
<diagram-server-type>kroki_io</diagram-server-type>
----

The default server is `https://kroki.komet.sh`. Override for local development:

[source,bash]
----
mvn verify -Dkroki.server.url=http://localhost:8000
----

Or with Docker:

[source,bash]
----
docker run -p 8000:8000 yuzutech/kroki
mvn verify -Dkroki.server.url=http://localhost:8000
----

Kroki supports 20+ diagram types including Mermaid, PlantUML, GraphViz, BPMN, Bytefield, Ditaa, Excalidraw, and more. Any diagram type supported by Kroki works in the pipeline.

=== SVGO Configurations

Two SVGO configs handle different SVG compatibility issues:

==== `svgo.config.mjs` (Prawn CSS Inlining)

Used by `build-all.sh` for the Prawn three-step build. Inlines CSS class rules from `<style>` blocks into element-level `style` attributes, then removes the empty `<style>` elements. This makes SVGs compatible with `prawn-svg`, which cannot parse CSS class selectors.

PlantUML and GraphViz SVGs do not use CSS classes, so they pass through unchanged.

==== `svgo-xep.config.mjs` (Full SVG Fix)

Used by the Prince, AH, and XEP profiles. Performs four transformations:

. *Inline CSS rules* -- same as above
. *Convert HSL to hex* -- Mermaid's default theme uses `hsl()` colors; XEP and some renderers do not support HSL in SVG
. *Replace `<foreignObject>` with `<text>`* -- extracts label text and creates native SVG `<text>` elements with computed positions
. *Fix ER diagram edges* -- removes the `undefined;undefined` prefix that Mermaid adds to ER relationship labels

=== Diagram Type Compatibility

PlantUML and GraphViz produce SVGs with:

* Native `<text>` elements for labels
* Hex color values (`#rrggbb`)
* No `<foreignObject>` elements
* No CSS class selectors

These SVGs work in *all* renderers without any svgo processing. Only Mermaid SVGs require the foreignObject-to-text conversion.

This compatibility difference is the primary reason IKE documentation standards prefer PlantUML and GraphViz over Mermaid. The svgo post-processing for Mermaid SVGs is lossy (text positioning is approximate after foreignObject-to-text conversion) and adds build complexity. PlantUML and GraphViz avoid this entirely.
